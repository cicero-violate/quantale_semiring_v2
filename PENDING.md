# Pending

Progressive gaps in the current system. Items here are forward improvements only — nothing that would require touching the active dispatch path without a clear replacement ready.

---

## 1. `CompiledCkaPattern.parallel_groups` is dead weight

**What:** `src/pattern.rs` compiles CKA `par` branches into both `edges` (tensor deltas) and `parallel_groups` (node-ID vectors). The `edges` are still consumed — they seed the tensor world at epoch start. The `parallel_groups` field is populated at every epoch but has no consumer since the CPU batch scheduler was deleted.

**Fix:** Remove `parallel_groups: Vec<Vec<i32>>` from `CompiledCkaPattern` and retire the parallel-group collection pass inside `compile_expr`. The CKA edge compiler and the par effect-safety validator in `crates/topology_core` must change together. The `par` algebra constraint (`validate_par_independence`) stays — only the output artifact changes.

**Risk:** Low. No runtime path reads `parallel_groups`. Tests that construct `CompiledCkaPattern` directly would need updating.

---

## 2. CKA pattern compilation belongs at build time

**What:** `load_default_patterns()` + `compile_pattern()` run at every epoch start (inside `build_runtime_epoch`). The result is a set of `TensorEdge` deltas that are merged into `tensor_edges` before constructing the world. The source file (`assets/patterns.source.json`) is already generated by `build-overlay`, so the runtime is re-doing work that build-overlay could do once.

**Fix:** Extend `build-overlay` to emit a `patterns.compiled.json` (or add the compiled edges directly into `topology.generated.json`). The runtime reads the pre-compiled edges instead of re-running the CKA compiler each epoch. `src/pattern.rs` becomes a build-time library only.

**Risk:** Medium. Requires coordinated changes to `topology_core/src/overlay.rs`, the asset loading path, and the epoch builder. The CKA compiler and tests are unchanged — only where it runs moves.

## 3. `TlogRecordKind::ExplorationExpand` and `append_exploration_expand` are never called

**What:** `tlog.rs` defines `TlogRecordKind::ExplorationExpand` and the method `TlogWriter::append_exploration_expand`, but nothing in the runtime calls `append_exploration_expand`. The record kind is written into the JSONL schema but produces no records.

**Fix:** Either wire `append_exploration_expand` to the expansion event in `main.rs` (exploration candidates are already logged at `ExplorationTopK`; this would add a per-token expansion record), or remove the variant and method entirely if the granularity is not wanted.

**Risk:** None — it is dead code. Removing it is a one-line change in `tlog.rs`.

---

## 4. Tensor feedback does not persist across epoch reloads

**What:** `load_learned_tensor_edges` reads `state/learned_edges.jsonl` to seed the tensor world with previously observed outcomes. However, the runtime loop has no write path back to this file. Successful execution receipts update the in-memory tensor but are never checkpointed. When an epoch reload happens (or the process restarts), all receipt-driven learning since the last manual checkpoint is lost.

**Fix:** After each successful `ProcessReceipt` that produces a non-trivial outcome, append the updated edge delta to `state/learned_edges.jsonl`. `LearningPolicy` already controls clamping and deduplication on load — the write side needs a symmetric `append_learned_edge(path, edge)` function and a call site in `main.rs` after `queue_execution_lattice_updates`.

**Risk:** Low for correctness (the existing load path already handles duplicates). The main consideration is write frequency — appending on every tick would grow the file quickly. Batching by epoch or by a configurable threshold is the right shape.

---

## 5. `TensorQuantaleWorld::project_parallel_group` / `commit_decision_batch` are unused API

**What:** `src/tensor.rs` still exposes `project_parallel_group` and `commit_decision_batch`, backed by the `tensor_quantale_project_batch` and `tensor_quantale_commit_batch` CUDA kernels in `quantale_world.cu`. These were the execution substrate for the CPU batch scheduler that was deleted. The CUDA kernel implementations are complete and correct.

**Fix (option A — clean up):** Remove the two methods from `TensorQuantaleWorld` and note in `quantale_world.cu` that the kernels are unused. This reduces the public API surface but discards a working CUDA capability.

**Fix (option B — future parallel dispatch):** Leave in place as the substrate for a future GPU-native parallel dispatch path that does not go through a CPU scheduler. The semantics are correct; what is missing is a dispatch policy that uses them from within `execute_active_node_blocking` or a new parallel commit tier in the loop.

The choice between A and B depends on whether parallel quantale commit is on the roadmap. If not, clean up.
