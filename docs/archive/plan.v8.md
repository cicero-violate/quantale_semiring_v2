# Plan v8: Fusion Proof, Runtime Smoke, and Paid-Problem Trace

## Objective

Prove that fused CUDA execution is behaviorally equivalent to the canonical
unfused tensor path, verify that receipts feed back into tensor learning, and
use one complete market-analysis trace to decide whether more fusion is worth
adding.

This plan is deliberately evidence-gated.  Fusion regions expand only after a
real repeated-compute profile shows the same CUDA-safe algebra is recurring and
material enough to optimize.

---

## Success Criteria

- Canonical and fused execution produce equivalent tensor outputs within a
  declared tolerance.
- CUDA feature detection has a runtime smoke test that can pass, skip, or fail
  explicitly.
- A receipt generated by a run changes the learned tensor feedback path in a
  testable way.
- Fused and unfused execution of the current market-analysis region are
  benchmarked with the same inputs and reported together.
- One full market-analysis trace is generated from feed ingestion through
  receipt, learning, and next tensor state.
- No new fusion region is added unless backed by repeated-compute evidence.
- The trace is attached to one recurring paid or manual problem where the agent
  can plausibly save human time.

---

## Work Items

### A  Canonical vs Fused Equivalence Test

Add a regression test that runs the existing market-analysis CUDA chain both
ways:

```text
Analysis::Return1
  -> Analysis::Volatility
  -> Analysis::SignalScore
```

The canonical path dispatches each operator independently.  The fused path uses
`topology.fusion.json` and `FusionDispatch` for the same region.  Both paths use
the same fixed input tensors for `market.price` and `market.open`.

Acceptance:

- Test fixture uses deterministic input tensors checked into `tests/fixtures/`
  or generated from fixed literals.
- Output comparison covers every externally visible write, especially
  `analysis.signal_score`.
- Intermediate-only tensors such as `analysis.return` and
  `analysis.volatility` may be checked in the canonical path for diagnostics,
  but they are not required as fused outputs.
- Tolerance is explicit, e.g. absolute error `<= 1e-5` for `f32`.
- The test skips cleanly when CUDA/NVRTC is unavailable, but never silently
  reports equivalence without running both paths.

Evidence artifact:

```text
tests/fusion_equivalence.rs
```

### B  CUDA Feature Runtime Smoke Test

Add a small smoke test that verifies the runtime can actually initialize the
CUDA feature path used by fusion.

The test should answer three questions separately:

1. Is CUDA support compiled in?
2. Is a CUDA device visible at runtime?
3. Can the JIT/fusion path compile and launch one minimal kernel?

Acceptance:

- A missing CUDA feature reports `skip: cuda feature disabled`.
- A missing device reports `skip: no cuda device`.
- Compile or launch failure is a hard failure when feature and device are both
  present.
- The smoke test is cheap enough to run before benchmark work.

Evidence artifact:

```text
tests/cuda_smoke.rs
```

---

### C  Receipt To Tensor Feedback Verification

Add a test or scripted trace that proves a receipt changes tensor learning.

Minimum loop:

```text
run decision/analysis step
  -> emit receipt
  -> load receipt through learning operator
  -> write learned edge delta
  -> reload/embed learned tensor edge
  -> observe changed transition score or projection rank
```

Acceptance:

- The receipt has a stable schema and includes enough node/edge identity to map
  back into the tensor graph.
- The learned edge update is visible in `state/learned_edges.jsonl` or an
  isolated test-state file.
- The before/after tensor edge value is asserted, not only logged.
- Negative receipts and positive receipts are handled distinctly if current
  policy supports both.

Evidence artifacts:

```text
tests/receipt_feedback.rs
state/traces/receipt_feedback_*.json
```

### D  Benchmark Fused vs Unfused Region

Add a benchmark command or test harness for the current fused market-analysis
region using identical inputs and repeated iterations.

Measure:

- Canonical operator-by-operator runtime.
- Fused region runtime.
- JIT compile time separately from steady-state launch time.
- Input length and iteration count.

Acceptance:

- Benchmark reports median or mean over repeated runs.
- First-run compile cost is not mixed into steady-state launch timing.
- Results include a speedup ratio and a note when the region is too small for
  fusion to matter.
- Benchmark can be run without mutating production state.

Evidence artifact:

```text
scripts/bench_fusion_region.sh
state/benchmarks/fusion_region_*.json
```

### E  Generate One Full Market-Analysis Trace

Generate one complete trace through the live or cached market-analysis loop:

```text
State::MarketFeed
  -> Analysis::Return1
  -> Analysis::Volatility
  -> Analysis::SignalScore
  -> State::TradeDecision
  -> State::PaperTrade
  -> receipt
  -> State::Learn
  -> updated tensor state
```

The trace may use cached market data if network access is unavailable.  It must
not place real trades or require broker credentials.

Acceptance:

- Trace includes input feed snapshot, chosen analysis region, tensor projection,
  paper decision, receipt, learned edge delta, and next selected node/rank.
- Every external effect is represented by a receipt or state file.
- Trace is reproducible from committed assets plus captured state.
- Failure paths are explicit; a provider failure trace is valid only if it still
  exercises receipt and learning feedback.

Evidence artifact:

```text
state/traces/market_analysis_full_*.json
```

### F  Increase Fusion Regions Only From Real Repeated Compute

Do not add fusion regions from intuition or graph shape alone.  Add a profiling
step that records repeated CUDA-safe compute sequences seen during real traces.

Candidate criteria:

- Same ordered kernel sequence appears in at least two traces or loop epochs.
- Region is already legal under the existing `Fusable(F)` constraints.
- Compute time or launch overhead is non-trivial relative to the whole loop.
- Region has deterministic tensor-only reads and writes.
- Equivalence test can be added before enabling the region by default.

Acceptance:

- Any proposed new fusion region is backed by a profile record.
- `topology.fusion.json` remains generated, not manually expanded.
- Boundary nodes, governance gates, host I/O, locks, and dynamic-shape nodes
  remain hard barriers.

Evidence artifacts:

```text
state/profiles/repeated_compute_*.json
docs/fusion-candidates.v8.md
```

### G  Attach To One Recurring Paid Or Manual Problem

Tie the trace to one concrete recurring workflow where the agent output can be
evaluated against a real human task.

Candidate problem:

```text
Daily market watchlist triage:
  ingest configured symbols,
  compute return/volatility/signal score,
  propose paper-only actions,
  produce a receipt-backed rationale,
  update routing based on observed decision quality.
```

Acceptance:

- The problem repeats on a schedule or on-demand manual trigger.
- The output replaces or shortens a real review step, not just a demo.
- The acceptance metric is observable, e.g. minutes saved per run, number of
  symbols triaged, or number of manual checks avoided.
- The workflow remains simulation-only until a separate plan introduces real
  broker integration and safety review.

Evidence artifact:

```text
docs/paid-problem-market-watchlist.v8.md
```

## Execution Order

1. CUDA smoke test.
2. Canonical vs fused equivalence test.
3. Fused vs unfused benchmark.
4. Receipt to tensor feedback verification.
5. Full market-analysis trace.
6. Repeated-compute profile extraction.
7. Paid/manual problem attachment.

The first two gates block all fusion expansion.  The benchmark and trace decide
whether more fusion is useful.  Receipt feedback and paid-problem attachment
decide whether the loop is worth recurring operational use.

---

## Non-Goals

- No real broker, exchange, wallet, or live order placement.
- No manual edits to generated fusion assets.
- No fusion of control flow, governance, boundary I/O, or host operators.
- No placeholder CUDA proof claims without runtime equivalence evidence.
- No new market-analysis operators unless the current trace shows a measured
  gap that existing operators cannot cover.

---
